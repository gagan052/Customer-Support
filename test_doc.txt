Chapter 4: Proposed System
Architecture and Design of the
Problem
4.1 Problem Definition
Build an AI-driven assistant that can:
1. answer user queries via a chatbot,
2. perform image understanding (object/text detection), and
3. assist with video analytics or basic video generation tasks.
The solution must run on Microsoft Azure, use Azure AI services, and expose APIs a Python
backend can consume.
4.2 Solution Overview
The system is a cloud-first, modular application with three primary capabilities:
• Conversational AI: An AI agent connected to a chatbot to interpret intents and respond.
• Computer Vision: Image detection, OCR, tagging, and descriptions.
• Video Intelligence: Indexing, scene insights, and basic assisted video creation or summaries.
All capabilities are orchestrated by a Python backend that calls Azure AI Services through secured
REST/SDK endpoints, persists results in storage, and logs telemetry.
4.3 High-Level Architecture
Key components
• Client Channels: Web app, Teams/Telegram (optional), or CLI.
• Bot & Orchestration: Azure Bot resource or direct Python webhook + AI agent invocations.
• Language Understanding: Azure language service for intent/entity extraction and grounding.
• Vision Services: Azure Computer Vision/OCR for image tasks.
• Video Services: Azure Video Indexer / Video Analyzer for video insights; optional templated
video creation pipeline.
• Python Backend (App Service/Container): Routing, business logic, validation, result
aggregation.
• Storage & Data: Azure Blob Storage (media), Azure Table/Cosmos DB (metadata), optional
Azure SQL (structured).
• Identity & Security: Azure AD, Managed Identity, Key Vault.
• Observability: Application Insights, Azure Monitor.
4.4 Component Design
• Frontend/Channel: Simple web UI or chat channel. Sends text or uploads media securely to
backend endpoints.
• Python Backend:
o Endpoints: /chat, /image/analyze, /video/analyze, /video/create.
o Tasks: Input validation, content moderation hooks, service selection, retry/circuit-
breaker, response formatting.
o SDK/REST integration: Calls Azure AI endpoints using Managed Identity or Key Vault-
stored secrets.
• Language Understanding: Extracts intent, entities, and generates assistant responses.
• Computer Vision: Object detection, OCR, description, tags.
• Video Intelligence: Uploads video to storage, triggers indexing, retrieves transcript, scenes,
faces, labels; optionally composes a highlight reel from timestamps.
• Data Layer:
o Blob Storage: Raw images/videos and analysis artifacts (JSON).
o Cosmos DB / Table: Per-request metadata, user session state, intent logs.
o SQL (optional): Reports, audit summaries, analytics-friendly tables.
• Security: Azure AD auth, RBAC, private endpoints where viable, Key Vault for secrets, content
moderation policy.
• Observability: Application Insights traces, metrics, request IDs, correlation IDs.
4.5 Data Design
• Entities
o UserSession(sessionId, userId, channel, startedAt, lastActiveAt)
o ChatTurn(sessionId, turnId, userText, intent, entities, botReply,
createdAt)
o ImageAnalysis(requestId, blobUrl, tags[], objects[], ocrText,
confidence, createdAt)
o VideoAnalysis(requestId, blobUrl, transcript, scenes[], people[],
labels[], keyMoments[], createdAt)
• Storage Strategy
o Large media in Blob; JSON analysis in Blob or Cosmos DB; compact metadata in
Cosmos DB/Table.
4.6 Process Flow (Text, Image, Video)
A. Chat/Text
1. User sends message → frontend.
2. Backend logs request, runs input checks.
3. Backend calls Language service for intent/entities.
4. If FAQ/KB, query KB; else route to vision/video if requested.
5. Compose reply and store ChatTurn.
6. Return response.
B. Image Analysis
1. User uploads image.
2. Backend stores in Blob, creates requestId.
3. Call Computer Vision/OCR → get tags, objects, text.
4. Persist results and return a summarized response.
C. Video Analysis / Creation
1. User uploads video or provides required inputs for creation.
2. Backend stores video and triggers Video Indexer/Analyzer.
3. Poll or event-driven callback for insights.
4. Persist insights; if creation requested, assemble segments/templates.
5. Return insights or generated artifact URL.

4.9 Deployment Topology
• App Service or Azure Container Apps for the Python backend.
• Private endpoints for AI services and Storage where feasible.
• Azure Front Door or Application Gateway in front of the API.
• Key Vault for secrets.
• Application Insights for telemetry.
4.10 Non-Functional Requirements
• Security: AAD auth, role-based access, encrypted storage in transit and at rest.
• Scalability: Autoscale backend and use async jobs for video.
• Reliability: Retry policies, idempotent request IDs, dead-letter for failed jobs.
• Performance: CDN for media delivery, caching of frequent KB responses.
• Observability: Centralized logs, metrics, and alerts.
• Cost Control: Tiered services, lifecycle rules for Blob, scheduled cleanup.
4.11 Assumptions and Constraints
• Internet access is available for cloud calls.
• Media sizes are within service limits.
• Some video features may be asynchronous and return results after analysis completes.
• Content moderation policies are enforced for uploads and outputs.
4.12 Traceability to Training Objectives
• AI Agents & Chatbot: Covered by the language service + bot orchestration.
• Image Detection: Computer Vision path and storage workflow.
• Video Creation/Analysis: Video pipeline with indexing and optional templating.
• Python Integration: Central orchestrator calling Azure AI via SDK/REST.

5.2 Module 1: AI Agent Integration with Azure Cognitive
Services
Objective:
To create an AI agent that can analyze user inputs and provide intelligent responses by utilizing Azure
Cognitive Services.
Implementation Steps:
1. Resource Creation:
o Created an Azure Cognitive Services resource and obtained API keys and endpoints.
o Configured services like Text Analytics, Language Understanding (LUIS), and QnA

o Developed a Python script using Azure SDK and REST API calls.
o Implemented logic to send user queries to Azure Cognitive Services for analysis and
response generation.
3. Features Implemented:
o Text Sentiment Analysis: Detects whether user input is positive, neutral, or negative.
o Language Detection: Automatically identifies the language of the given input.
o Entity Recognition: Extracts keywords, names, and places from text using Azure’s NLP
models.
4. Output:
The AI Agent analyzed user input and returned a structured response containing sentiment,
detected entities, and possible suggestions.
Example Python Snippet:
from azure.ai.textanalytics import TextAnalyticsClient
from azure.core.credentials import AzureKeyCredential
endpoint = "https://<your-resource>.cognitiveservices.azure.com/"
key = "<your-key>"
client = TextAnalyticsClient(endpoint=endpoint, credential=AzureKeyCredential(key))
text = ["Azure AI Services make it easy to build intelligent apps!"]
response = client.analyze_sentiment(documents=text)[0]
print("Sentiment:", response.sentiment)
print("Positive Score:", response.confidence_scores.positive)

Azure Speech Services, making interactions more natural and accessible.

6.3 Limitations
While the current system effectively demonstrates the core functionality of Azure AI Services, certain
limitations were observed during its implementation due to technical and resource constraints.
1. Limited Scope of Training Models
• The AI models used (LUIS, Computer Vision, etc.) were pre-trained and offered limited
customization.
• There was no opportunity to train domain-specific models due to time and data constraints.
2. Dependence on Internet and Cloud Services
• Since Azure AI operates in the cloud, the system requires continuous internet connectivity.
• Any network or service outage can interrupt AI operations.
3. Restricted Free-Tier Resources
• The Azure free-tier used during training imposed limits on API calls, storage, and computational
resources.
• High-volume tasks like large-scale video analysis or real-time processing could not be tested
extensively.
4. Simplified Data Handling
• The system used static or sample datasets for demonstration.
• Integration with live or large datasets would require enhanced data management, security, and
optimization techniques.
5. Lack of Deep Learning Customization
• Due to the scope of the training, participants primarily used pre-built AI services rather than
designing or training deep learning models manually.
• Future projects could involve model customization using Azure Machine Learning Studio.
6. Limited Real-Time Processing
• The prototype processed inputs (text, image, video) asynchronously.
• Real-time streaming capabilities for video or voice data were not implemented due to time
limitations.
7. Minimal UI/UX Design
• The chatbot and interface were kept minimal for demonstration purposes.
• Future iterations can include an advanced front-end design with improved interactivity and
accessibility features.
8. Cost Constraints for Long-Term Usage
• Continuous use of Azure services beyond the free tier can become expensive for individual users
or small teams without sponsorship or institutional support.
